# Design: Absorb the admin panel into the main viewer

**Date:** 2026-06-24
**Status:** Approved (pending spec review)

## Summary

The bottom-bar buttons (`+ Experiment`, `+ Protocol`, `+ Report`) currently link to a
separate `admin.html` page that duplicates browsing the main catalogue and is the only
place create/edit forms live. This redesign **absorbs that admin panel into the main
viewer**: create/edit happens inline via modal overlays injected onto the catalogue
pages, `admin.html`/`admin.js` are deleted, reports gain a git-backed version selector,
and Publish is wired to rebuild `catalog/` so the unified view never goes stale.

## Goals

- Editing and creating experiments, protocols, and documents happens **in place** on the
  catalogue pages â€” no navigation to a separate admin page.
- The admin panel (`catalog/admin.html`, `catalog/admin.js`) is removed entirely.
- Reports get a **version indicator + selector** driven by git history (one version per
  Publish commit).
- Every Publish rebuilds `catalog/` so the local view tab is never stale after an
  edit-then-publish (the bug that triggered the Publish-refresh handoff).

## Non-goals

- No versioning UI for **code** pages (that would be re-implementing GitHub).
- No **Add** button for **reports** (they are autogenerated from experiments).
- The **export** bundle (`eln/share.py â†’ export_all`) stays a separate manual action and
  is not merged into Publish.
- No revert/restore button on the reports version selector â€” viewing past versions only;
  restoring is git's job.

## Background â€” current architecture

Three regeneration/output surfaces are currently independent:

- **publish** (`eln/server/publish.py â†’ publish()`): materialize `CODE-NN`
  (`allocate_experiment_codes`) â†’ dump DB to `experiments.sql` â†’ `git add` the tracked
  `PUBLISH_PATHS` â†’ commit â†’ push. It does **not** rebuild `catalog/`. Its docstring
  still claims "GitLab CI rebuilds the catalog from experiments.sql" â€” stale; there is no
  CI anymore.
- **catalog/**: gitignored static pages built by `generate_all` (`eln/generators/`). This
  is what the local admin server renders as its view. Refreshed by `labbook regenerate`
  (`cmd_regenerate`) or the server's `/api/regenerate` endpoint (`eln/server/app.py`).
- **export** (`eln/share.py â†’ export_all`, `cmd_export`): builds a standalone shareable
  bundle to a manual `--dest`; internally calls `generate_all(root, catalog_out=dest)`.
  Separate manual action.

Relevant facts confirmed during design:

- `reports/` **is** committed by Publish (it is in `PUBLISH_PATHS`), so report files have
  real git history â€” one snapshot per Publish commit.
- `catalog/*.html` is gitignored/derived; `catalog/admin.html` and `catalog/sdgl.html`
  are currently committed source files (`.gitignore` has `!catalog/admin.html`).
- Edit buttons today (`catalog/edit-overlay.js`) all navigate to `/admin.html?edit=...`;
  the admin page **is** the editor. Experiments/protocols/reports have an Edit button;
  experiments/protocols/reports have create forms in the admin panel.
- **Documents have no editor at all**: no Edit button, no Add, no admin form, and **no
  `/api/documents` endpoints**. They are read-only pages scanning `ROOT/documents/` for
  markdown/notebook files (`eln/generators/documents.py`), no DB involved â€” structurally
  identical to reports.

## Approach

The chosen approach (over an iframe embed or server-rendered hidden forms) is to
**extract a shared `forms.js` module**: pull the form markup and behaviors out of
`admin.js`/`admin.html` into one module that `edit-overlay.js` injects into a modal
overlay. This gives true inline editing with a single source of truth and lets
`admin.html` be deleted. Cost: a real refactor of the ~1000-line `admin.js`.

## Design

### 1. Editor architecture

- New `catalog/forms.js` exporting:
  - `openExperimentForm(id?)` â€” create when `id` omitted, edit otherwise.
  - `openProtocolForm(id?)`
  - `openDocumentForm(path?)`
  - `openReportEditor(path)` â€” edit only.
- Each function builds its form into a **modal overlay** (`<dialog>`-based, large and
  scrollable since the experiment form is tall), wires the existing behaviors (tag/cell
  chips, microscopy channel rows, codeâ†”title bidirectional autofill, autocomplete
  datalists from `/api/field-values`), and submits to the **same `/api/*` endpoints that
  exist today**. No endpoint changes for experiments/protocols/reports.
- `edit-overlay.js` stops linking to `admin.html`. Edit/Add buttons call the `forms.js`
  functions instead of navigating.
- **Delete** `catalog/admin.html` and `catalog/admin.js`. Remove the
  `!catalog/admin.html` exception from `.gitignore`. Remove the server route that serves
  `admin.html` if one exists, and any admin-only wiring that becomes dead.

### 2. Per-surface Edit/Add behavior

- **Experiments** (`experiments.html`): keep the per-row **Edit** button, but it opens
  the modal instead of navigating. Add an **+ Add experiment** button in the page header.
- **Protocols** (`protocols.html`): **Edit** opens the modal; add **+ Add protocol**. The
  existing in-DB protocol version selector is unchanged.
- **Documents** (`documents.html`): add a per-card **Edit** button and an **+ Add
  document** button, backed by **new `/api/documents` endpoints** (see Â§5). Write
  `.md`/`.ipynb` files to `ROOT/documents/`.
- **Reports** (`reports.html`): **Edit** opens the modal (markdown content, or
  notebook-markdown-cells only, exactly as today). **No Add** button.

### 3. Reports version selector (git-backed, local-server only)

- Each report card gains a **version indicator** (e.g. "Published v3 Â· 2026-06-20") and a
  **dropdown** listing prior Publish commits that touched that report file.
- New endpoints:
  - `GET /api/reports/<path:filename>/versions` â€” list commits touching the file (via
    `eln/analysis/gitref.py`), newest first, with short sha + date + commit subject.
  - `GET /api/reports/<path:filename>?version=<sha>` â€” return the rendered report from
    that historical git blob, read-only.
- Selecting an older version re-renders it in place, read-only. The indicator/selector
  **only functions in the live server view** (it needs git + the server). The
  static/exported `reports.html` shows the latest version with the selector absent or
  inert â€” it degrades gracefully.
- A version = a Publish commit (the granularity the user chose). Local-only edits that
  haven't been published are the "current/working" view, not a numbered version.

### 4. Freshness wiring (Publish-refresh handoff)

- **Inline save:** after any successful create/edit/delete, `forms.js` calls
  `/api/regenerate` and then reloads (or re-fetches the affected fragment), so the local
  view reflects the change immediately â€” even before publishing. This fixes staleness at
  the edit source, not just at Publish.
- **Publish:** `publish()` in `eln/server/publish.py` calls `generate_all(root)` after
  `allocate_experiment_codes` and around the `experiments.sql` dump, so `catalog/` is
  rebuilt on every Publish.
  - `catalog/` stays gitignored and uncommitted (consistent with the "derived files not
    committed" rule).
  - Publish keeps its git backup/tracking role unchanged (commit/push
    `experiments.sql` + the rest of `PUBLISH_PATHS`).
  - Fix the stale "GitLab CI rebuilds the catalog from experiments.sql" reference in the
    `publish.py` docstring.

### 5. New documents endpoints

Clone the existing reports endpoints in `eln/server/app.py`, pointed at `ROOT/documents/`
instead of `ROOT/reports/`:

- `GET /api/documents` â€” list documents (path, title).
- `GET /api/documents/<path:filename>` â€” return markdown content, or notebook markdown
  cells (reuse `_read_notebook_markdown_cells`).
- `POST /api/documents` â€” create a markdown document under `ROOT/documents/`.
- `PUT /api/documents/<path:filename>` â€” overwrite markdown, or apply markdown-cell edits
  for notebooks (reuse `_apply_markdown_cell_edits`; code cells/outputs preserved).
- `DELETE /api/documents/<path:filename>` â€” delete the document file.

Reuse the reports path-resolution/safety helper (the documents analogue of
`_resolve_report_path`) so writes are confined to `ROOT/documents/`.

### 6. Toolbar

- Remove the `+ Experiment`, `+ Protocol`, `+ Report` links from the bottom bar in
  `edit-overlay.js` â€” Add now lives contextually on each catalogue page.
- Keep **Publish** and **Export catalog**.

## Data flow

- **Create/edit (e.g. experiment):** click `+ Add experiment` / `Edit` â†’ `forms.js` opens
  modal â†’ submit â†’ `POST`/`PUT /api/experiments` â†’ on success `forms.js` calls
  `/api/regenerate` â†’ reload â†’ updated row visible immediately.
- **Publish:** click Publish â†’ `POST /api/publish` â†’ `publish()` runs
  `allocate_experiment_codes` â†’ `generate_all(root)` (rebuilds `catalog/`) â†’ dump
  `experiments.sql` â†’ `git add`/commit/push `PUBLISH_PATHS` â†’ toast success. View is fresh.
- **View report history:** open a report â†’ version dropdown calls
  `GET /api/reports/<file>/versions` â†’ pick a commit â†’ `GET â€¦?version=<sha>` returns the
  rendered historical blob â†’ shown read-only.

## Testing

- **publish:** extend `tests/server/test_publish.py` to assert `generate_all` is invoked
  (e.g. `catalog/experiments.html` reflects a freshly-added tag after `publish()`), and
  that `catalog/` remains uncommitted/gitignored.
- **documents endpoints:** new tests mirroring `tests/server/test_app.py` reports cases â€”

- **reports versions:** test `GET /api/reports/<file>/versions` returns commits in order

  file has only one commit.
- **generators:** ensure existing generator tests still pass with the new markup hooks;
  add coverage for the documents page Edit/Add hooks if markup changes.
- Manual: edit an experiment's tags inline â†’ save â†’ tags appear in view without Publish;
